Configuración de DNS con BIND9

Servidor DNS maestro/esclavo, zonas directas e inversas, resolución recursiva segura.

⏱ 3h ⚡ Dificultad: Alta BIND9DNSLinux

00Introducción

Configurarás un servidor DNS autoritativo maestro/esclavo con BIND9 sobre Ubuntu/Debian: zonas directas (nombre→IP) e inversas (IP→nombre), transferencias de zona seguras con clave TSIG y resolución recursiva controlada por ACL.

Escenario

RolHostnameIP
DNS Maestrons1.lab.local192.168.10.10
DNS Esclavons2.lab.local192.168.10.11
Dominiolab.local—
DNS es la base de casi todo: Active Directory, correo, web... Un DNS mal configurado provoca fallos difíciles de diagnosticar.

01Instalación de BIND9

En ambos servidores (maestro y esclavo):

bash
sudo apt update
sudo apt install bind9 bind9utils bind9-dnsutils -y
named -v        # versión
sudo systemctl enable --now named

Archivos principales en /etc/bind/:

  • named.conf.options — opciones globales (recursión, forwarders).
  • named.conf.local — definición de zonas.
  • db.* — ficheros de zona.

02Opciones globales y recursión segura

Edita named.conf.options en el maestro. Definimos una ACL para limitar quién puede hacer consultas recursivas:

bash
sudo nano /etc/bind/named.conf.options
named.conf.options
acl "redes-internas" {
    127.0.0.1;
    192.168.10.0/24;
};

options {
    directory "/var/cache/bind";

    recursion yes;
    allow-query     { redes-internas; };
    allow-recursion { redes-internas; };

    forwarders { 1.1.1.1; 9.9.9.9; };
    forward only;

    dnssec-validation auto;
    listen-on { 127.0.0.1; 192.168.10.10; };
    listen-on-v6 { none; };

    version "no disponible";   // oculta la versión
};
Seguridad. Restringir allow-recursion evita que tu DNS sea usado en ataques de amplificación DDoS. Nunca dejes recursión abierta a Internet.

03Clave TSIG para transferencias

Las transferencias de zona maestro→esclavo deben autenticarse con una clave compartida TSIG.

bash
tsig-keygen -a HMAC-SHA256 transfer-key > /tmp/transfer.key
cat /tmp/transfer.key

Copia el bloque resultante a /etc/bind/named.conf.local en ambos servidores:

named.conf.local
key "transfer-key" {
    algorithm hmac-sha256;
    secret "PEGAR_AQUI_EL_SECRETO_GENERADO==";
};

04Zonas en el maestro

Declara las zonas directa e inversa en named.conf.local del maestro:

named.conf.local
zone "lab.local" {
    type master;
    file "/etc/bind/zones/db.lab.local";
    allow-transfer { key "transfer-key"; };
    also-notify { 192.168.10.11; };
};

zone "10.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/zones/db.192.168.10";
    allow-transfer { key "transfer-key"; };
    also-notify { 192.168.10.11; };
};

Fichero de zona directa

bash
sudo mkdir -p /etc/bind/zones
sudo nano /etc/bind/zones/db.lab.local
db.lab.local
$TTL 3600
@   IN  SOA ns1.lab.local. admin.lab.local. (
        2026062401  ; Serial (AAAAMMDDnn)
        3600        ; Refresh
        1800        ; Retry
        1209600     ; Expire
        86400 )     ; Negative TTL
;
@       IN  NS  ns1.lab.local.
@       IN  NS  ns2.lab.local.
;
ns1     IN  A   192.168.10.10
ns2     IN  A   192.168.10.11
srv     IN  A   192.168.10.20
www     IN  CNAME srv
@       IN  MX 10 mail.lab.local.
mail    IN  A   192.168.10.25

Fichero de zona inversa

db.192.168.10
$TTL 3600
@   IN  SOA ns1.lab.local. admin.lab.local. (
        2026062401 3600 1800 1209600 86400 )
;
@       IN  NS  ns1.lab.local.
@       IN  NS  ns2.lab.local.
;
10      IN  PTR ns1.lab.local.
11      IN  PTR ns2.lab.local.
20      IN  PTR srv.lab.local.
25      IN  PTR mail.lab.local.
Regla de oro. Incrementa SIEMPRE el serial al editar una zona; si no, el esclavo no detectará cambios.

05Configuración del esclavo

En el esclavo (ns2), declara las zonas como type slave apuntando al maestro:

named.conf.local (esclavo)
zone "lab.local" {
    type slave;
    file "/var/cache/bind/db.lab.local";
    masters { 192.168.10.10 key "transfer-key"; };
};

zone "10.168.192.in-addr.arpa" {
    type slave;
    file "/var/cache/bind/db.192.168.10";
    masters { 192.168.10.10 key "transfer-key"; };
};
El esclavo no tiene ficheros de zona propios: los recibe del maestro por transferencia AXFR/IXFR.

06Validar y arrancar

Comprueba la sintaxis antes de recargar, en ambos nodos:

bash
sudo named-checkconf
sudo named-checkzone lab.local /etc/bind/zones/db.lab.local
sudo named-checkzone 10.168.192.in-addr.arpa /etc/bind/zones/db.192.168.10
sudo systemctl reload named
sudo journalctl -u named -n 30 --no-pager

07Pruebas de resolución

Desde un cliente apuntando a 192.168.10.10:

bash
dig @192.168.10.10 www.lab.local +short
dig @192.168.10.10 srv.lab.local A
dig @192.168.10.10 -x 192.168.10.20 +short     # inversa
dig @192.168.10.11 www.lab.local +short        # contra el esclavo
# Transferencia (debe fallar sin clave => seguridad OK):
dig @192.168.10.10 lab.local AXFR
  • Si la directa e inversa resuelven en ambos nodos: replicación correcta.
  • Si AXFR sin clave es denegada: transferencias protegidas.
  • Revisa que el serial coincide en maestro y esclavo.
Para diagnóstico: sudo rndc status y sudo rndc zonestatus lab.local.
