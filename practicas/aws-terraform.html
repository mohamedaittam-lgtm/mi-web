Infraestructura en AWS con Terraform

IaC: VPC, subredes, EC2, S3 y security groups versionados en Git.

⏱ 5h ⚡ Dificultad: Alta AWSTerraformIaC

00Introducción

Con Terraform definirás infraestructura como código (IaC) en AWS: una VPC con subredes pública y privada, una instancia EC2, un bucket S3 y security groups, todo versionado en Git y reproducible con terraform apply.

Aviso de costes. Esta práctica crea recursos que pueden generar coste. Usa la capa gratuita (t2.micro/t3.micro) y ejecuta terraform destroy al terminar.

01Requisitos y credenciales

  • Cuenta de AWS y un usuario IAM con permisos (evita la cuenta root).
  • Terraform ≥ 1.5 instalado.
  • AWS CLI configurado.
bash
# Instalar Terraform (Ubuntu):
wget -O- https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform -y
terraform version

# Configurar credenciales AWS:
aws configure   # Access Key, Secret, región eu-west-1
Seguridad. Nunca subas tus claves a Git. Usa variables de entorno o ~/.aws/credentials y añade *.tfvars y .terraform/ a .gitignore.

02Estructura del proyecto

Organiza el código en ficheros separados por responsabilidad:

árbol de directorios

03Proveedor y variables

provider.tf
terraform {
  required_version = ">= 1.5"
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}
provider "aws" {
  region = var.region
}
variables.tf
variable "region"     { default = "eu-west-1" }
variable "project"    { default = "asir-lab" }
variable "vpc_cidr"   { default = "10.0.0.0/16" }
variable "az"         { default = "eu-west-1a" }
variable "key_name"   { description = "Nombre del key pair EC2" }
variable "my_ip"      { description = "Tu IP /32 para SSH" }

04VPC, subredes y routing

network.tf
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  tags = { Name = "${var.project}-vpc" }
}

resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "${var.project}-igw" }
}

resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = var.az
  map_public_ip_on_launch = true
  tags = { Name = "${var.project}-public" }
}

resource "aws_subnet" "private" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.2.0/24"
  availability_zone = var.az
  tags = { Name = "${var.project}-private" }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
  tags = { Name = "${var.project}-rt-public" }
}

resource "aws_route_table_association" "public" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public.id
}

05Security groups y EC2

compute.tf
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"] # Canonical
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }
}

resource "aws_security_group" "web" {
  name   = "${var.project}-web-sg"
  vpc_id = aws_vpc.main.id

  ingress {
    description = "SSH solo desde mi IP"
    from_port = 22 ; to_port = 22 ; protocol = "tcp"
    cidr_blocks = [var.my_ip]
  }
  ingress {
    description = "HTTP"
    from_port = 80 ; to_port = 80 ; protocol = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port = 0 ; to_port = 0 ; protocol = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
  tags = { Name = "${var.project}-web-sg" }
}

resource "aws_instance" "web" {
  ami                    = data.aws_ami.ubuntu.id
  instance_type          = "t3.micro"
  subnet_id              = aws_subnet.public.id
  vpc_security_group_ids = [aws_security_group.web.id]
  key_name               = var.key_name
  user_data = <<-EOF
    #!/bin/bash
    apt-get update -y && apt-get install -y nginx
    echo "<h1>ASIR · IaC con Terraform</h1>" > /var/www/html/index.html
  EOF
  tags = { Name = "${var.project}-web" }
}

06S3 y outputs

storage.tf
resource "aws_s3_bucket" "data" {
  bucket = "${var.project}-data-${random_id.suffix.hex}"
  tags   = { Name = "${var.project}-data" }
}
resource "random_id" "suffix" { byte_length = 4 }

resource "aws_s3_bucket_versioning" "data" {
  bucket = aws_s3_bucket.data.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_public_access_block" "data" {
  bucket                  = aws_s3_bucket.data.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}
outputs.tf
output "web_public_ip" { value = aws_instance.web.public_ip }
output "bucket_name"   { value = aws_s3_bucket.data.bucket }

07Despliegue y ciclo de vida

El flujo de trabajo de Terraform:

bash
terraform init        # descarga el provider
terraform fmt         # formatea el código
terraform validate    # valida la sintaxis
terraform plan        # previsualiza cambios
terraform apply       # aplica (confirma con 'yes')

# Comprueba la web:
curl http://$(terraform output -raw web_public_ip)

# Al terminar, destruye TODO para no pagar:
terraform destroy

Buenas prácticas IaC

  • Versiona el código en Git con ramas y PRs.
  • Usa un backend remoto (S3 + DynamoDB) para el state compartido y con lock.
  • Separa entornos (dev/prod) con workspaces o carpetas.
  • Nunca edites recursos a mano en la consola (drift).
Backend remoto. Para el state remoto: backend "s3" { bucket="..." key="infra.tfstate" dynamodb_table="tf-lock" }
