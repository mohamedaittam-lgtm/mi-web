Cluster MySQL con replicación maestro-esclavo

Replicación asíncrona, backups automatizados y monitoreo básico.

⏱ 3h 30m ⚡ Dificultad: Alta MySQLLinuxBash

00Introducción

Montarás una replicación asíncrona maestro→esclavo en MySQL 8 sobre dos servidores Ubuntu, con backups automatizados mediante mysqldump + cron y una comprobación básica de monitoreo del estado de la réplica.

RolHostnameIPserver-id
Maestrodb1192.168.20.101
Esclavodb2192.168.20.112
La replicación da alta disponibilidad de lectura y una copia caliente de los datos, pero NO sustituye a los backups.

01Instalar MySQL en ambos nodos

bash
sudo apt update
sudo apt install mysql-server -y
sudo systemctl enable --now mysql
sudo mysql_secure_installation

Permite conexiones de red en ambos: edita bind-address.

bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
mysqld.cnf
bind-address = 0.0.0.0

02Configurar el maestro

Activa el binary log y asigna un server-id único:

bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
mysqld.cnf (maestro)
[mysqld]
server-id           = 1
log_bin             = /var/log/mysql/mysql-bin.log
binlog_format       = ROW
binlog_expire_logs_seconds = 604800   # 7 días
# (opcional) replicar solo una BD:
# binlog_do_db      = tienda
bash
sudo systemctl restart mysql

Crear usuario de replicación

bash
sudo mysql
SQL (maestro)
CREATE USER 'repl'@'192.168.20.11' IDENTIFIED WITH caching_sha2_password BY 'ClaveFuerte#2026';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'192.168.20.11';
FLUSH PRIVILEGES;
SHOW MASTER STATUS\G
Apunta esto. Anota File (ej. mysql-bin.000003) y Position del SHOW MASTER STATUS: los necesitarás en el esclavo.

03Volcado inicial de datos

Para que el esclavo parta del mismo estado, vuelca los datos del maestro de forma consistente:

maestro
# En el maestro:
mysqldump -u root -p --all-databases --master-data=2 --single-transaction \
  --routines --triggers > /tmp/dump.sql
# Copia al esclavo:
scp /tmp/dump.sql asir@192.168.20.11:/tmp/
esclavo
# En el esclavo:
mysql -u root -p < /tmp/dump.sql
--master-data=2 añade como comentario la posición exacta del binlog, útil para verificar el CHANGE MASTER.

04Configurar el esclavo

bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
mysqld.cnf (esclavo)
[mysqld]
server-id  = 2
relay_log  = /var/log/mysql/mysql-relay-bin.log
read_only  = ON
super_read_only = ON
bash
sudo systemctl restart mysql
sudo mysql

Apuntar al maestro e iniciar la réplica

SQL (esclavo)
CHANGE REPLICATION SOURCE TO
  SOURCE_HOST='192.168.20.10',
  SOURCE_USER='repl',
  SOURCE_PASSWORD='ClaveFuerte#2026',
  SOURCE_LOG_FILE='mysql-bin.000003',
  SOURCE_LOG_POS=1234,
  GET_SOURCE_PUBLIC_KEY=1;
START REPLICA;
SHOW REPLICA STATUS\G
En MySQL 8.0.22+ se usa la sintaxis REPLICA/SOURCE. En versiones previas: SLAVE/MASTER (CHANGE MASTER TO, START SLAVE, SHOW SLAVE STATUS).

05Verificar la replicación

En SHOW REPLICA STATUS\G busca estos valores:

  • Replica_IO_Running: Yes
  • Replica_SQL_Running: Yes
  • Seconds_Behind_Source: 0 (o bajo)
  • Last_Error vacío.

Prueba funcional

maestro
-- En el maestro:
CREATE DATABASE tienda;
USE tienda;
CREATE TABLE clientes(id INT PRIMARY KEY, nombre VARCHAR(50));
INSERT INTO clientes VALUES (1,'Ada');
esclavo
-- En el esclavo (debe aparecer ya):
USE tienda; SELECT * FROM clientes;

06Backups automatizados

Script de backup con rotación, ideal para ejecutar en el esclavo y no penalizar al maestro:

bash
sudo nano /usr/local/bin/mysql-backup.sh
mysql-backup.sh
#!/usr/bin/env bash
set -euo pipefail
DEST=/var/backups/mysql
DAY=$(date +%F)
mkdir -p "$DEST"
mysqldump -u backup -p'ClaveBackup#2026' --single-transaction \
  --all-databases | gzip > "$DEST/full-$DAY.sql.gz"
# Conservar 14 días:
find "$DEST" -name 'full-*.sql.gz' -mtime +14 -delete
echo "Backup OK: $DEST/full-$DAY.sql.gz"
bash
sudo chmod 700 /usr/local/bin/mysql-backup.sh
# Programar a las 03:15 cada día:
echo '15 3 * * * root /usr/local/bin/mysql-backup.sh' | sudo tee /etc/cron.d/mysql-backup
Crea un usuario backup con privilegios mínimos: GRANT SELECT, LOCK TABLES, SHOW VIEW, EVENT, TRIGGER ON *.* TO 'backup'@'localhost';

07Monitoreo básico

Script que alerta si la réplica se detiene o se retrasa:

check-replica.sh
#!/usr/bin/env bash
S=$(mysql -u root -p'***' -e "SHOW REPLICA STATUS\G")
io=$(grep 'Replica_IO_Running:' <<<"$S" | awk '{print $2}')
sql=$(grep 'Replica_SQL_Running:' <<<"$S" | awk '{print $2}')
lag=$(grep 'Seconds_Behind_Source:' <<<"$S" | awk '{print $2}')
if [ "$io" != "Yes" ] || [ "$sql" != "Yes" ]; then
  echo "ALERTA: replicación detenida (IO=$io SQL=$sql)"
elif [ "$lag" -gt 60 ]; then
  echo "ALERTA: retraso de $lag s"
else
  echo "OK: réplica al día (lag=$lag s)"
fi
Siguiente nivel. Integra esta comprobación con la práctica de Prometheus/Grafana usando el mysqld_exporter para dashboards y alertas reales.
